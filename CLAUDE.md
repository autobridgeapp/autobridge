# AutoBridge — standing rules

These apply to every session working in this repo.

1. **RLS on every table, on creation.** Any new Supabase table gets `alter table ... enable row level security;` plus explicit policies in the same migration that creates it. Never leave a table with RLS disabled or with no policies (which blocks all access) as a "come back to it later" step.
2. **`service_role` key is server-only.** It must never appear in client-side code, never be read via a `NEXT_PUBLIC_*` env var, and never ship to the browser. It only ever lives in a server-side env var (e.g. `SUPABASE_SERVICE_ROLE_KEY`, no `NEXT_PUBLIC_` prefix), read only from server components, route handlers, or server actions.
3. **New UI matches [design-reference/autobridge-prototype-v2.jsx](design-reference/autobridge-prototype-v2.jsx).** Use its tokens: bg `#F5F5F2`, ink `#101112`, accent `#FF4400`, fit-green `#00A868`, line `#E4E4DF`, card `#fff`; Archivo (display/mono weight 900 italic for wordmarks) + JetBrains Mono for labels/badges. Match its screens and component patterns (mobile-frame shell, bottom tab nav, card grid, pill filters) rather than introducing new visual language.
4. **Fitment is always computed dynamically, never hardcoded.** "Does this listing fit my car" is derived at request/render time from `listing_fitment` rows plus the signed-in user's primary `garage_vehicles` row (see `lib/fitment.ts`). Never reintroduce a static reference vehicle (like the old `MY_CAR` constant) or a precomputed `fits_my_car`-style boolean column — a user's garage can change, so the match must be recomputed every time, not stored.
5. **New listings default to universal fitment.** If a seller doesn't specify fitment when creating/editing a listing, insert a `universal = true` row rather than leaving the listing with no fitment rows at all — an empty fitment set means it silently disappears from every "fits my car" filtered view.
6. **Realtime tables need participant-scoped RLS, not public read.** Any table added to the `supabase_realtime` publication (e.g. `messages`) must have a row-scoped `select` policy (participant/owner-only). Supabase Realtime enforces each subscriber's RLS on `postgres_changes`, so a public-read policy on a realtime table broadcasts every row to every connected client, not just the intended participants (e.g. the two people in a thread).
7. **Mutations run client-side, never during Server Component render.** Every insert/update/delete (likes, messages, listings, garage, etc.) goes through the browser Supabase client inside a `"use client"` component, then `router.refresh()` or a realtime event picks up the change. Server Components only read data. Don't call `.insert()`/`.update()`/`.delete()` inside a Server Component's render just to prep data for display — keep reads and writes on separate sides of that boundary so RLS always runs as the signed-in user, not the server session.
